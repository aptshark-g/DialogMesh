## §0 自适应原则：参数设计的通用范式

本文档中每个模块都涉及阈值、权重、系数等参数。这些参数的设计遵循统一的三层范式：

### 0.1 三层范式

**第一层：文献锚点**
从公开基准/论文中查到参数的参考值，作为初始值。这个值不是"最优值"，是"合理起点"。
锚点的作用是给参数一个不依赖直觉的初始位置——即使不准，也有据可查。

**第二层：安全区间**
每个参数设一个浮动范围。区间的下界是"低于这个值，系统功能退化"，上界是"高于这个值，系统能力过剩但浪费资源"。
区间的意义不是"最优解在这之间"，而是"在这个范围内，系统不会崩溃"。

**第三层：语义对齐自适应**
参数随实际使用数据调整。调整信号**必须和参数的语义作用对象对齐**，而不是使用全局信号（如用户满意率）。

| 参数 | 语义对齐信号 | 为什么是这个信号 |
|------|-------------|----------------|
| 编译器 confidence 阈值 | LLM vs 规则在本维度的 confidence 差值 | 差值指示谁在该维度更可靠 |
| 语义邻居 cosine 阈值 | BGE 在 MTEB 上的实际相关度分布 | 相关度分布决定了邻居范围 |
| 预测器四维权重 | 各维度对命中率变化趋势的贡献 | 趋势指示哪个维度在帮倒忙 |
| BehaviorGraph α/β/γ/δ | 各因子的样本量（不是质量） | β 随 N 增长自然提升 |
| 奖励数值 | 权重波动幅度 | 剧烈波动说明奖励值过大 |
| FoA decay | 子图大小（自监督） | 子图 <2 表示 decay 过高，>5 表示过低 |

### 0.2 为什么需要这个范式

本文档中大部分参数无法通过理论推导得到精确值——Stanza 的中文 UAS 不是 95% 而是 87-89%，BGE 的 MTEB 相似度分布受语料影响，四维权重没有学术文献可参考。

传统做法就是"拍脑袋 → 上线看 → 凭感觉调"。这个范式给了一个替代方案：**锚点给方向，区间给安全网，自适应给个性化。**

### 0.3 实现时的参数配置格式

每个工程实现文档应包含一个参数配置表，格式如下：

| 参数 | 初始值 | 锚点来源 | 区间 | 自适应信号 | 自适应速率 |
|------|--------|---------|------|-----------|-----------|
| compiler.threshold | 0.75 | Stanza UAS (87-89%) | [0.65, 0.85] | LLM vs rule confidence diff | ±0.02/次 |
| embed.neighbor_threshold | 0.6 | BGE MTEB | [0.50, 0.70] | 相似度分布 | ±0.01/次 |
| predict.w_llm | 0.40 | 无文献 | [0.25, 0.55] | 命中率趋势 | ±0.02/10轮 |
| reward.top1_hit | +0.10 | 无文献 | [+0.05, +0.20] | 权重波动 | ±0.01/100轮 |
| graph.decay | 0.3 | ACT-R 文献 | [0.2, 0.5] | 子图大小 | ±0.02/50轮 |


## 1. 约束补全编译器

### 要解决什么问题

将自然语言输入映射为 `{语义槽位, 置信度, 稳定性}` 的结构化表示。核心挑战是**LLM不做显式句法解析**，而纯规则方案面临**约束消解的组合爆炸**。

### 算法选择：二合一流水线

**不是"LLM和规则两边跑然后比较结果"，而是"LLM做粗切割，规则只在LLM不确定的维度上做功"。**

### 三步流水线

#### Step 1: LLM 伪拆解（粗切割）

**输入**: `sentence: str` + `context: ParseContext`（包含前序轮次的实体、话题、约束状态）

**LLM 输出格式**（约 100 tokens）:
```
{
  "slots": {
    "agent":     {"value": "人",       "confidence": 0.95},
    "action":    {"value": "喝",       "confidence": 0.98},
    "patient":   {"value": "饮料",     "confidence": 0.90},
    "result":    {"value": "呛",       "confidence": 0.95},
    "cause":     {"value": "碳酸刺激", "confidence": 0.60}
  },
  "utterance_type": "statement",
  "sentiment": "neutral"
}
```

**Prompt 设计关键**:
- 明确指示 LLM **不需要推理原因**，只需结构化回显
- 每个槽位必须同时输出 `value` 和 `confidence`
- `confidence` 的定义是"LLM 对自己这个判断有多确定"，不是"这个槽位的客观可信度"

**为什么不是让 LLM 直接画语法树**:
- LLM 不天然做依存句法分析（训练目标不是这个）
- NLP 领域 SOTA 的依存句法解析（如 Stanza）在对话短文本上准确率也不到 90%
- 让 LLM 做"语块→语义角色"的粗粒度标注，比"词→依存弧"更匹配 LLM 的能力边界

#### Step 2: 规则选择性深挖

**关键洞察**: 不对所有维度做功，只对 `confidence < 0.75` 的维度进行规则消解。

**为什么是 0.75**: Stanza 的 UAS 约 95%，说明 NLP 工具在确定性约束上能做到 90%+ 准确率但不到 100%。0.75 以上认为 LLM 已经足够确定，规则引擎不需要介入——避免规则引擎的错误覆盖 LLM 的正确判断。

**深挖算法**（针对单个不确定维度）:
1. **约束框架匹配**: 从预置的语义框架库（FrameNet 精简版）中查找当前维度对应的约束集合。例如 `cause("呛") → {碳酸, 辛辣, 气管误入, 过敏}`
2. **跨维度交叉验证**: 检查已确定维度的值是否与候选约束冲突。例如 `patient="饮料"` 排除了 `辛辣` 和 `过敏`（域不兼容）。
3. **上下文约束补充**: 如果对话前序提到"自制酵素饮料"，则将 `发酵气体` 加入候选。如果没提到，优先高频候选（汽水→碳酸）。
4. **剩余候选排序**: 按约束交叉验证的命中数排序，top-1 分配置信度 0.82。

**复杂度控制**: 约束空间原本 5×10×20=1000 种组合，现在只针对单个维度做功（~3 种候选），<3ms。

#### Step 3: 稳定性评分

```python
stability = mean(confidence_i) × (1 - variance(confidence_i))
# 所有槽位置信度高 → stability 高
# 存在低置信度槽位（高方差） → stability 被拉低
# stability < 0.6 → 编译器输出"不确定"，交由上层处理
```

### 流式验证

在 LLM 流式输出过程中，每收到一条 JSON key-value，立刻比对约束规则库。硬冲突（如 `agent="汽车"` 而 `frame("喝")` 要求 agent 为生命体）→ 标记该维度 → LLM 输出完成后用规则结果覆盖。

**为什么不断 LLM 流**: 中断流会导致 token 浪费（已生成的 token 白算）。标记+后覆盖是最小成本的矫正方式。

### 降级路径

```
LLM 可用     → 完整三步流水线
LLM 不可用   → 规则四步: 语法分解(jieba+词性标注) →
               约束激活(语义框架匹配) →
               约束消解(交叉验证) →
               稳定性评分
LLM 超时     → 同上 + 标记 degraded=true
```

### 关键边界

| 场景 | 处理 |
|------|------|
| LLM 输出非 JSON | 重试一次，仍失败→降级规则路径 |
| 所有维度 confidence > 0.75 | 跳过 Step 2，直接算 stability |
| 所有维度 confidence < 0.75 | 规则全量消解，跳过 LLM 置信度（不可靠） |
| stability < 0.6 | 输出 `{undefined: true}` + 原因列表，不做硬猜测 |

## 2. 行为语义嵌入层

### 要解决什么问题

BehaviorGraph 的边权重基于精确字面匹配。当用户执行新的行为对（如从未见过的"运行脚本→查API文档"），`freq_ratio=0`，只能退回到 LLM 的 `causal_prob`——波动大、成本高。

### 算法选择：谓词-论元原型嵌入

**替代方案对比**:

| 方案 | 优势 | 劣势 |
|------|------|------|
| 纯 BGE-small 嵌入 | 简单，通用 | "运行程序"vs"删除程序" cosine=0.78 灾难 |
| 动词-名词加权解耦 | 工程折中 | 需要手动维护 10+ 类型的 (w_v, w_n) 权重表 |
| **谓词-论元原型嵌入** | 谓词类提供硬边界 | 需要 <50 条映射规则 |

**选择谓词-论元的原因**: 动词-名词解耦仍依赖 BGE-small 嵌入向量做余弦相似度，没有从根本上区分谓词（动词语义是质变而非量变）。谓词-论元用离散的谓词类（execute/delete/analyze…）提供了硬边界。

### 算法流程

**离线预计算**（一次性）:
1. 定义 <50 个常见谓词类: `{execute, analyze, modify, delete, monitor, create, stop, check, config, navigate, ...}`
2. 为每个谓词类构造**原型向量**: 用 BGE-small 编码该类的标准表述（如 "execute program" 和 "run service"），取平均作为该类的原型向量。
3. 定义论元类: `{Program, Service, Script, Log, Code, Result, Report, Config, File, Parameter, Metric, Project, ...}`
4. 论元保留 BGE-small 的完整语义编码能力（不过论元类，直接用原始嵌入）。

**在线推理**（每轮，<5ms）:
1. **谓词-论元拆分**: LLM 轻量调用（~50 tokens）将 `action_summary` 拆分为 `(谓词, 论元)`。例如"运行程序" → `("运行", "程序")`。
2. **谓词分类**: 将"运行"映射到谓词类 `execute`（规则匹配，<1ms）。
3. **论元嵌入**: 用 BGE-small 编码"程序"→ 384 维向量。
4. **谓词原型检索**: 直接取 `execute` 的原型向量（查表，O(1)）。
5. **加权组合**: `vec = w_pred × prototype_vec(pred_class) + w_arg × bge(arg)`

**`w_pred` 和 `w_arg` 的取值**（与 v3.3 设计一致）:

| 行为类型 | 谓词权重 | 论元权重 | 原因 |
|---------|---------|---------|------|
| TOOL_EXEC | 0.7 | 0.3 | 执行/删除本质区别 |
| CODE_RUN | 0.7 | 0.3 | 同上 |
| LOG_CHECK | 0.4 | 0.6 | 看什么比怎么看重要 |
| ENTITY_ANALYZE | 0.3 | 0.7 | 分析对象决定方向 |
| CONFIG_MODIFY | 0.5 | 0.5 | 均重要 |
| EXPLORATION | 0.3 | 0.7 | 领域决定方向 |

### 相似度计算

```python
sim = cosine_similarity(vec_a, vec_b)
# execute(Program) vs execute(Script) → 谓词相同, 论元接近 → sim ~0.72
# execute(Program) vs delete(Program) → 谓词不同, 论元相同 → sim ~0.35
# execute(Program) vs analyze(Log)    → 谓词不同, 论元不同 → sim ~0.12
```

### 三层权重查询

BehaviorGraph 查询 `from_action → to_action` 的边权重时:

```
Step 1: 精确匹配 → 在 BehaviorGraph 中按字面查找 → 命中 → 直接返回
Step 2: 语义邻居 → Top-K 相似行为对（谓词-论元嵌入, cosine > 0.6）
         → 加权平均所有匹配边的权重（权重按相似度加权）
Step 3: 全局退避 → 返回 None → BehaviorPredictor 用 LLM causal_prob
```

**为什么用 0.6 作为语义邻居阈值**:
- BGE-small 在 MTEB 上的平均相似度约 0.55-0.65 为人可感知的"相关"
- 0.6 既能过滤掉无关行为，又能保留语义邻居（"运行程序"→"启动服务" = ~0.68）
- 阈值太低 → 噪声邻居污染权重；太高 → 邻居太少，退回到 LLM

### 关键边界

| 场景 | 处理 |
|------|------|
| 谓词分类失败（不在 <50 条规则中） | 退回到纯 BGE-small 嵌入，不拆分 |
| 精确匹配命中 + 语义邻居也命中 | 精确匹配权重 × 0.8 + 邻居加权平均 × 0.2 |
| 语义邻居 Top-K 全低于 0.6 | 等同于 Step 3 退避 |
| 索引需重建时机 | 新的行为对超过总行为对的 20% → 触发重建 |

### 附录: 谓词-论元嵌入的演进路径

**v3.1.1 动词-名词解耦**:
  vec = w_v * bge(verb) + w_n * bge(noun)
  问题: BGE-small 对中文动词区分度不够。
  [运行程序] vs [删除程序] 动词部分 cosine ~0.55

**v3.3 谓词-论元嵌入**:
  谓词类(execute/delete/modify)用离散分类提供硬边界

  - 动词-名词解耦本质上是[信任 BGE 能区分动词],

  - 而 v3.1.1 动词-名词解耦下 sim ~0.78 (灾难)

## 3. BehaviorGraph

### 要解决什么问题

记录用户行为的序列关系和因果权重，支持"给定当前行为，预测下一步最可能做什么"。

### 数据结构

**不是一般图而是带标签有向加权图**，节点是 `BehaviorStep`，边是 `BehaviorEdge`。

- **节点**: 轻量（~5 个字段），通过 `step_id` 全局索引。存储行为摘要、类型、结果、时间戳。
- **边**: 带权重，权重由四项组成（见下文）。存储统计计数和最后更新时间。

关键设计决策：**TopicTreeNode 只存储行为链的拓扑结构**（节点+边 ID），**BehaviorGraph 是独立的图引擎**，存储边权重。两者通过 `edge_id` 引用，不重复存储权重。

### 权重更新算法：指数移动平均 + 增量修正

```
新权重 = α × LLM因果概率 + β × 频率比 + γ × 画像增强 + δ × 结构先验 + (1-α-β-γ-δ) × 旧权重
```

**各项的含义和来源**:

| 项 | 来源 | 更新频率 | 取值范围 |
|----|------|---------|---------|
| LLM因果概率 | BehaviorPredictor 的 LLM 推理 | 每轮（如果有 LLM 预测） | [0, 1] |
| 频率比 | `sample_count / (sample_count + correction_count)` | 每轮 | [0, 1] |
| 画像增强 | 用户认知画像匹配度 | 准实时（每 N 轮） | [0, 0.3] |
| 结构先验 | 因果基地元角色映射 | 离线更新 | [0, 0.7] |

**初始 α、β、γ、δ 的取值建议**: α=0.25, β=0.30, γ=0.05, δ=0.05, 余量 0.35 → EMA 平滑。

**为什么不是简单频率统计（贝叶斯更新）**: 纯频率统计需要独立同分布假设，但行为序列明显不是——后面的行为依赖于前面的结果。用 EMA 允许近期行为有更大权重，适应行为漂移。

### 快速纠正通道

当同一行为对连续被用户纠正 2 次以上时:
```
跳过 EMA 平滑 → 直接设置为 w_new = 0.3 × w_old（大幅降低）
或 w_new = 0.5 × reasonable_alternative（如果用户指定了正确行为）
```

之后由 Meta-Cognitive 做慢调整恢复正常更新。

**为什么需要这个**: EMA 平滑在低 α 下收敛太慢。用户纠正 3 次，EMA 可能只降了 40%，用户感知是"纠正了三次还记不住"。

### 稀疏性处理

用户行为空间高维稀疏 → 大多数行为对的 `freq_ratio = 0` 且 `sample_count = 0`。

**处理链**:
1. 语义嵌入层邻居查询（见 §2）
2. 邻居也无数据 → LLM 因果概率（BehaviorPredictor）
3. LLM 也不确定 → 冷启动种子（<15 个手工行为对）
4. 冷启动也不匹配 → 均匀分布（相当于"我不知道"）

**冷启动种子的维护**: 不是一次性写入后不管了。每 50 轮对话触发 Reflective 检查：种子行为对的 `sample_count` 超过 10 → 用实际数据权重替代种子权重。种子数据不再需要的 → 标记为 `deprecated` 并从查询路径中移除。

### 关键边界

| 场景 | 处理 |
|------|------|
| 行为对从未出现 | 语义邻居 → LLM → 种子 → 均匀分布 |
| `sample_count > 100` 但 `structural_prior = 0` | 触发 LightweightCausalDiscovery |
| `correction_count / sample_count > 0.5` | 标记为不稳定边，降低融合器的预测置信度 |
| 图节点数超过 10000 | 触发不活跃节点裁剪（30天未激活的叶子节点） |

## 4. BehaviorPredictor

### 要解决什么问题

给定当前行为链 + 用户画像 + BehaviorGraph 权重，预测用户下一步最可能的行为 Top-N。

### 算法选择：LLM 推理生成候选 + 四维价值排序

**为什么不用序列模型（LSTM/Transformer）**: 冷启动时期样本不足（需要数千轮对话才够训练），且用户行为模式个体差异大（一个 embedding 无法覆盖不同用户的调试风格）。

**为什么不用马尔可夫链**: k 阶马尔可夫在 k>2 时状态空间爆炸，且只能基于精确行为匹配，没有语义泛化能力。

### 两步算法

#### Step 1: LLM 生成候选行为

**输入**: `behavior_chain_summary`（最近 5 步的摘要） + `user_profile`（调试风格、常见错误模式、偏好工具）

**Prompt 策略**:
- 让 LLM 生成 **3-5 个候选行为** + 每个的因果概率
- Prompt 中注入 BehaviorGraph 的 Top-3 高权重后继 → LLM 以此为参考但不被限制
- 因果概率必须解码为数值

**为什么不让 LLM 直接做最终排名**: LLM 的概率估计受 prompt 措辞影响极大（同样的语义换种说法概率差 30%）。LLM 负责"提出可能性"，价值排序负责"判断好坏"。

#### Step 2: 四维价值排序

对每个候选行为计算期望价值:

```python
expected_value = (
    llm_probability * 0.4              # LLM 估计的因果概率
    + outcome_success_rate * 0.3       # 历史上该行为的成功率
    + (1 - cognitive_load) * 0.2       # 认知负载惩罚（越复杂排名越低）
    + user_profile_match * 0.1         # 与用户画像的匹配度
)
```

**各项的具体计算**:

1. **LLM 因果概率** (0.4): 直接来自 Step 1 LLM 输出。作为种子权重，不是最终决策。
2. **历史成功率** (0.3): `BehaviorGraph.success_count / (success_count + failure_count)`。如果是新行为（统计为 0），取 0.5（中性先验）。
3. **认知负载** (0.2): 根据行为类型预定义复杂度。`TOOL_EXEC=0.2`, `CODE_RUN=0.3`, `LOG_CHECK=0.1`, `ENTITY_ANALYZE=0.4`, `EXPLORATION=0.3`。用户越累越应该推荐简单操作。
4. **画像匹配** (0.1): 从用户认知画像中获取偏好行为模式。例如"喜欢先看日志再改代码"的用户，`LOG_CHECK` 的行为得分会提升。

**排序后输出**: Top-3 候选 + 期望价值 + 各维度分解（用于可解释性）。

### 训练闭环

```
预测 [A(0.73), B(0.15), C(0.08)]
用户实际做了 C

如果 C 在 Top-3 内 → 预测方向对，reward = +0.05
如果 C 不在 Top-3 → 预测失败，reward = -0.15
如果 C 是 Top-1 → 精准预测，reward = +0.10

reward 传给 BehaviorRewarder → 更新 BehaviorGraph 边权重
```

### 关键边界

| 场景 | 处理 |
|------|------|
| Position 1: LLM 可用但 BehaviorGraph 无数据 | 只用 LLM causal_prob（权重分布: 0.7 LLM + 0.3 画像） |
| Position 2: BehaviorGraph 有数据但 LLM 不可用 | 只用 BehaviorGraph 权重 + 画像匹配 |
| Position 3: 都不可用 | 返回 Top-5 全局高频行为（类似冷启动种子） |
| 候选行为质量低（所有 `expected_value < 0.3`） | 返回 `["ask_clarification"]` — 建议系统向用户提问 |

## 5. BehaviorRewarder

### 要解决什么问题

用户做出实际行为后，计算奖励信号并更新 BehaviorGraph 的边权重。核心挑战是**区分有价值的纠正信号和无价值的噪声**。

### 奖励规则表

| 场景 | 奖励值 | 说明 |
|------|--------|------|
| 预测 Top-1 命中 | +0.10 | |
| 预测 Top-3 命中 | +0.05 | 方向对但不够精准 |
| 预测实体部分匹配 | +0.03 | 行为不对但实体对 |
| 预测完全失败 | -0.15 | |
| 用户明确纠正 | -0.20 | 比预测失败更严重的惩罚 |
| 用户纠正 + 指定正确行为 | -0.20 + 提升正确路径权重 | |
| 用户无反馈 | 0.00 | 不更新 |

### 时间衰减因子

```python
decay = exp(-Δt / τ)

τ = 300s  当 30s < Δt ≤ 5min   # 温和衰减
τ = 3600s 当 Δt > 60min        # 显著衰减
无衰减    当 Δt ≤ 30s          # 快速连续操作保持强关联
```

**为什么用指数衰减而不是阶梯衰减**: 行为间的因果关联随时间是连续的信号衰减，不是阶跃的。用 `exp(-Δt/τ)` 比 `if Δt < 5min then 1.0 else 0.5` 更精确地反映因果链的自然衰减。

### 噪声自适应

**关键设计决策**: 上线初期不做自适应，统一 `noise_level = 0.5`。攒 2-4 周真实数据后，分析:
1. 纠正是否在 3 轮内被用户再次纠正（自我矛盾 → 噪声）
2. 纠正后是否持续沿用新路径（有效学习 → 信号）
3. 纠正文本中是否有"不对"/"不是"/"重新来"等明确否定词（强化信号）

噪声级别影响的是单轮奖励的权重，而不是过滤掉信号。高噪声用户 → `effective_reward = raw_reward × 0.3`（降低单轮影响），观察窗口从 3 轮拉长到 7 轮。

### ABL 反思向量（v3.3 新增）

除了数值奖励，还生成结构化反思:

```python
{
  "error_type": "over_generalization" | "missing_step" | "wrong_entity" | "domain_mismatch",
  "correct_path": "分析日志",
  "why_wrong": "运行程序→修复代码在调试初期不成立，因为还没确认报错原因",
  "suggested_correction": "在修复代码前插入'分析日志'作为中间步骤"
}
```

反思向量不直接影响边权重（reward 已经做了），但写入 Cognitive Tree 的 REFLECTION 节点，供 Meta-Cognitive 做跨轮次模式分析。

### 关键边界

| 场景 | 处理 |
|------|------|
| 预测失败 + 用户沉默 + 2 轮后重复同样行为 | 可能是系统预测对了只是用户没反馈 → 轻微正向偏置 |
| 同一路径连续纠正 3 次 | 标记为"持久错误"，跳过 EMA 直接硬覆盖 |
| 用户行为是 EXPLORATION 类型 | Exploration 的默认奖励值取 0（探索性行为不参与训练） |
| session 结束 | 触发一次全局权重衰减（所有边 × 0.95），模拟长期记忆衰减 |

## 6. 融合器

### 要解决什么问题

四轨（算法、LLM、预测、因果基地）各自产出不同维度的结果，需要合并为单轮决策。关键挑战是**四轨返回时间差一个数量级**，且各轨结果可能在数值尺度和可信度上不可直接比较。

### 算法选择：分阶段融合 + 简化版全局工作空间

### 分阶段融合（时序对齐）

**问题**: Track-0 10ms、Track-1 150ms、Track-P 80ms、因果基地 <1ms。如果等到最慢的回来再融合，延迟 150ms。

**方案**: 分三个阶段融合，每个阶段产生一个中间结果。

```
T=10ms    Stage 1: 算法 + 因果基地
          输入: Track-0 的 {entities, params, hard_facts} + Causal 的 {structural_priors}
          输出: 硬事实融合结果（实体、参数、因果约束）
          策略: STRUCTURAL > FACTUAL，两者都不需要语义理解，可以立即合并

T=80ms    Stage 2: + 预测
          输入: Stage 1 结果 + Track-P 的 {predicted_actions, expected_outcomes}
          输出: 预加载注入（在 Stage 1 结果上附加预测信息）
          策略: 预测结果不改变 Stage 1 的决策，只作为预加载工具和 context bias

T=150ms   Stage 3: + LLM 语义
          输入: Stage 2 结果 + Track-1 的 {intent, semantics, implicit_needs}
          输出: 最终决策
          策略: SEMANTIC 层补充+修正。如果 Stage 1 和 Stage 3 冲突 → 见冲突仲裁

降级:
T=150ms 内 Track-1 未返回 → Stage 2 作为最终输出
T=80ms 内 Track-P 未返回 → 跳过 Stage 2
T=10ms 内 Track-0 未返回 → 降级到纯 LLM 决策
```

**为什么不是"全部到位→一次性融合"**: 时序对齐是实时系统的基本约束。分批融合让系统在 10ms 内就有可用输出（硬事实层），150ms 内做语义增强。如果 LLM 挂了，前两层仍然能给出靠谱的决策。

### 五级冲突优先级

```
STRUCTURAL  >  FACTUAL  >  SEMANTIC  >  STRATEGIC  >  PREDICTIVE
因果基地判定  算法实体提取  LLM语义理解  保留候选方案   预测结果

STRUCTURAL 最高 → 因果基地判定基于形式化证明，不可覆盖
PREDICTIVE 最低 → 预测是"猜测"，不能覆盖已确认的事实
```

### 简化版全局工作空间

**不是完整实现 Baars 理论**，而是在融合器上加 ~100 行的**有状态轨道竞争**:

```python
track_repression_count = {"algo": 0, "llm": 0, "pred": 0, "causal": 0}

每轮融合时:
1. 基础优先级 = ["causal", "algo", "llm", "strategic", "pred"]
2. 被连续压制 > 3 次的轨道 → 优先级 +1（强制广播一次）
3. 选择优先级最高且 confidence > 0.5 的轨道输出
4. 被选中的轨道 → reset count = 0
5. 未被选中的轨道 → count += 1
```

**为什么只要 100 行就够了**: 完整的全局工作空间需要并行处理器竞争+广播+意识闸门，工程复杂度极高。简化版只做"有状态的轨道优先级调整"，本质上是一个 `dict + priority_queue`，但解决了"某个轨道一直被压制"的公平性问题。

### 主动矛盾消解

Track-P 和 Track-1 在融合前对比:

| 矛盾类型 | 检测方式 | 处理 |
|---------|---------|------|
| INTENT_ACTION_MISMATCH | Track-1 的 intent 和 Track-P 的 predicted_action 语义不同 | 降低当前决策置信度 + 预加载预测的工具 |
| ENTITY_PREDICTION_MISMATCH | 两轨的实体列表不同 | 触发实体二次提取 + 标记"需澄清" |
| CONFIDENCE_DIVERGENCE | 两轨 confidence 都高但结论不同 | 同时降低两者权重 + 强制保守路径 |

**为什么矛盾不等到 Meta-Cognitive 才处理**: 矛盾是**紧急信号**——说明系统当前认知模型可能有问题。立即降级比"先输出再说、后面复盘"更安全。

### 关键边界

| 场景 | 处理 |
|------|------|
| Stage 1 输出没有语义理解 | 降级到 Stage 2/3 补充 |
| 四轨全返回 confidence < 0.5 | 输出 `ask_clarification` 而不是硬猜 |
| Track-1 和因果基地在 STRUCTURAL 层冲突 | 置信度比较: 因果 > 0.9 → 因果胜; < 0.9 → 两者都标记并降级保守路径 |
| 用户 Profile = LITE | 只跑 Stage 1（双轨: 算法+LLM），跳过预测和因果 |

## 7. 因果基地元角色

### 要解决什么问题

当 BehaviorGraph 没有历史数据、LLM 因果概率不可靠时，提供第三种知识来源：结构因果先验。

### 算法选择：三步映射

**为什么不是一步映射**: 自然语言→元角色骨架的直接映射如果出错，`structural_prior` 不是先验而是"错误的必然"。

### 三步映射算法

#### Step 1: 自然语言 → 约束条件

由约束补全编译器完成（不是新步骤）。从编译器输出中提取:

```python
constraints = {
    "domain_hint": "thermal+electronic",  # 物理域/电子域/软件域/...
    "has_feedback": False,
    "involves_dissipation": True,         # 是否有耗散
    "involves_storage": False,            # 是否有存储
    "causal_direction": "cause→effect",   # 因果方向
    "involves_transformation": True       # 是否有状态转换
}
```

#### Step 2: 约束条件 → 候选结构列表

用约束条件在元角色骨架空间中搜索。**不是穷举搜索**——约束条件将搜索空间从 8^5=32768 缩小到 ~10-20 个候选。

**约束→骨架的匹配规则**:
```
involves_dissipation=True → 骨架必须包含 dissipate 角色
has_feedback=True         → 骨架必须包含环（非 DAG）
involves_storage=True     → 骨架必须包含 store 角色
causal_direction=单向      → 骨架中不能有 junction_split 同时有 cycle
```

**候选排序**: 对候选骨架计算约束满足度（满足约束数/总约束数），按满足度降序排列。

#### Step 3: 匹配度排序 → structural_prior

```python
if score > 0.8:   structural_prior = 0.7   # 高置信，但不说是"必然"
if 0.5 < score ≤ 0.8: structural_prior = 0.3   # 中等置信
if score ≤ 0.5:   structural_prior = 0.0   # 不采用
```

**关键设计**: `structural_prior` 上限是 0.7，永远不说是"确定性因果"。这来自设计文档的三步匹配修正——约束条件驱动的匹配度，不是因果基地的判定。

### 元角色枚举（8 个角色）

| 角色 | 语义 | 物理域 | 软件域 | 组织域 |
|------|------|--------|--------|--------|
| source | 能量/信息/资源的源头 | 电源、用户输入 | API 调用、事件触发器 | 需求方 |
| sink | 能量/信息/资源的终点 | 散热器、负载 | UI 渲染、文件写入 | 最终用户 |
| store:potential | 势变量存储 | 电容、弹簧 | 缓存、状态变量 | 预算、配额 |
| store:kinetic | 流变量存储 | 电感、质量 | 消息队列、缓冲区 | 在制品库存 |
| dissipate | 能量耗散/信息损失 | 电阻、摩擦 | 速率限制、超时丢弃 | 流程审批延迟 |
| transform | 能量/信息形式转换 | 变压器、齿轮 | 序列化器、格式转换 | 处理工序 |
| junction_sum | 多输入→一输出汇聚 | 并联→串联 | 多信号合并 | 多部门→统一决策 |
| junction_split | 一输入→多输出分发 | 串联→并联 | 广播、fan-out | 统一部署→多点执行 |

### `structural_prior` 何时写入 BehaviorEdge

1. **触发条件**: 行为链长度超过 10 步 → 触发元角色映射（离线）。
2. **映射过程**: 行为链的每个 `action_type + entities` → 三步映射 → 候选骨架。
3. **写入**: 如果某条边的两端行为都成功映射到元角色骨架，且骨架的匹配度 > 0.5 → 写入 `structural_prior`。
4. **δ 的动态调整**: 初始 δ = 0.05。每个 Reflective 周期检查: 如果 `structural_prior > 0.3` 且 `correction_count` 低（用户不纠正这条边），上调 δ 至 0.15。如果 `correction_count` 高，维持 δ = 0.05 或下调。

### 关键边界

| 场景 | 处理 |
|------|------|
| 约束条件不完整（编译器 stability < 0.6） | 跳过映射 → structural_prior = 0 |
| 候选骨架匹配度最高 < 0.5 | 跳过 → structural_prior = 0 |
| 行为链长度 < 5 | 不触发离线映射（太短不构成可分析的因果链） |
| 候选骨架有 2+ 个得分接近 | 全部保留，取平均 structural_prior, 标记 multi_match |

## 8. 负知识库

### 要解决什么问题

形式化知识说"这不可能"时，防止系统执行有害行为。核心挑战是**"不可能"有时候不是绝对的**——纯水在常温不导电，但高压电解时导电。

### 算法选择：三级分类 + 上下文开关 + 三次熔断

### 三级分类

| 级别 | 含义 | 触发后行为 | 可被用户覆盖？ |
|------|------|-----------|-------------|
| HARD_BLOCK | 物理/数学证明不可能 | 直接拦截 + 解释 | 否 |
| WARN | 常态下不该做，有已知特例 | 拦截一次 + 警告 + 解释 | 是（3 次后放过） |
| SOFT_DISCOURAGE | 统计上不可行 | 不过滤，只降低融合器权重 | 是 |

### HARD_BLOCK 的验证：do-calculus

HARD_BLOCK 不靠人工标注，而是用 do-calculus 的**后门准则**验证（见 §11）。只有 do-calculus 验证通过 OR 键合图/Petri 网形式化证明的关系，才能打 HARD_BLOCK 标签。

### 三次熔断

```
WARN 级别:
第 1 次拦截 → BLOCK + 解释（如"纯水在常温下不导电"）
第 2 次拦截 → WARN（不拦截, 但加警告）("注意：你正在尝试不寻常的操作")
第 3 次拦截 → 自动降级为 SOFT_DISCOURAGE + ALLOW
  用户坚持 3 次 → 系统学习新的上下文规则
  → 记录: "当用户明确表示'电解'场景时，纯水导电规则不适用"
```

### 上下文开关

每条规则带 `condition: Callable[[Context], bool]`:

```python
rule = ContextualNegativeRule(
    condition=lambda ctx: "电解" not in ctx.topic_keywords
                          and "高压" not in ctx.topic_keywords,
    level=WARN,
    message="纯水在常温下不导电。如果你在分析电解场景，这条规则不适用。"
)
```

### 关键边界

| 场景 | 处理 |
|------|------|
| 没上下文的规则 → 全局生效 | 不用 condition |
| 规则冲突（WARN 和 HARD 同时触发） | HARD_BLOCK 覆盖 |
| 规则被降级为 ALLOW_AND_LEARN | 学习新上下文规则 → 写入用户画像的 domain_exceptions |

## 9. FoA 注意力焦点

### 要解决什么问题

L2Summary 是完整的 BehaviorDAG（可能几百个节点），当前轮只需要最相关的 3-5 个节点。全量加载浪费上下文窗口且引入噪声。

### 算法选择：ACT-R 双权重激活传播

**为什么是 ACT-R**: ACT-R 的激活传播模型天然适合"以当前焦点为种子、沿关联边传播激活、自然衰减"的场景。且不需要训练，只需要设定两个参数：关联权重和衰减率。

### 算法流程

**每轮触发**（在 L1Summary 生成后）:

1. **种子选择**: `current_intent` + `pcr_expectation` 作为初始种子节点（1-2 个）。
2. **激活传播**:
   ```
   a(node) = base_activation(node)
           + Σ(weight(edge) × a(source))  # 从种子节点沿边传播
           - decay × dist_from_seed(node)  # 距离衰减
   
   base_activation: 节点的重要性（由 L2Summary 中该节点的度数决定）
   weight(edge): BehaviorEdge 的当前权重
   decay: 默认 0.3
   ```
3. **阈值裁剪**: 只保留 `a(node) > 0.3` 的节点。
4. **子图输出**: 3-5 个节点 + 它们之间的边 → 注入上下文窗口的 Hot 区。

### 示例

用户说"那个报错还记得吗？":

1. 种子 = "报错"（ERROR_TRIGGERED 事件）
2. 传播: "报错" → (weight 0.82) → "查看日志" → (weight 0.65) → "分析堆栈"
3. 激活: "报错"=1.0, "查看日志"=0.82, "分析堆栈"=0.53, 其他节点 < 0.3
4. 输出子图: {"报错", "查看日志", "分析堆栈"}
5. 系统直接用这 3 个行为节点回答，无需 LLM 搜索整个摘要

### 关键边界

| 场景 | 处理 |
|------|------|
| 种子节点不在 L2Summary | 回退到语义搜索（按 action_summary 找最相似的行为节点） |
| 传播后 0 个节点 > 0.3 | 降级到 LLM 搜索完整摘要 |
| 多个种子节点（多意图） | 分别传播，取并集 |
| 激活太高（>0.9 的节点太多） | 强制取 Top-5（窗口限制） |

## 10. L1Summary

### 要解决什么问题

每轮对话后生成结构化摘要，供后续检索和二级摘要聚合。核心约束：不能丢失行为链/因果链/关联链的信息。

### 算法：三级自适应

```
输入: Turn（原始轮次文本 + 编译器输出 + 行为链当前状态）

Step 1: 内容分类 → 决定用哪个策略
  - 确定性内容 (工具输出/代码/明确指令) → 规则引擎
  - 半确定性内容 (明确指令/参数) → 模板填充
  - 非确定性内容 (模糊需求/情感) → LLM 语义压缩

Step 2: 按策略生成核心语义
deterministic: core_semantics = format(turn.action_type, turn.entities, turn.result)
                # 例: "执行 scan_memory 0x1000 → 返回 0x00"
template:      core_semantics = template(intent, entities, params)
                # 例: "用户要求修改 PID 为 1234 的进程配置"
llm:           core_semantics = llm_compress(turn.raw_text, max_chars=100)
                # 例: "用户感觉饮料太呛，猜测是碳酸过多"

Step 3: 元信息提取（三种策略都要做）
meta_info = {
    "prev_action": 上一轮的 action_type,
    "current_action": 本轮的 action_type,
    "predicted_next": BehaviorPredictor 的 Top-3,
    "causal_events": 本轮是否触发了 ERROR_TRIGGERED / USER_CORRECTION 等,
    "associations": 关联链引用（跨主题实体引用）,
    "is_topic_switch": 是否切换话题,
    "user_satisfaction": 从 sentiment 和 follow-up behavior 估计,
    "correction_detected": 是否检测到纠正,
}

Step 4: 存储（不阻塞下一轮）
→ 写入 L1Summary 表 + 更新双向索引
→ 检查是否需要触发 L2Summary 重新聚合（同一话题新增摘要 > 5 条 → 触发）
```

### 压缩比

| 策略 | 信息丢失 | 延迟 | 软目标 |
|------|---------|------|--------|
| 规则引擎 | 0% | <1ms | 原文保留 |
| 模板填充 | ≤10% | 1-3ms | 保证实体/参数完整 |
| LLM 语义压缩 | ≤20%（软目标） | 50-200ms | 不丢失行为意图 |

### 为什么 LLM 压缩容忍 20% 丢失

行为和因果信息已经在 `meta_info` 中结构化保存了（零丢失）。LLM 压缩的 `core_semantics` 字段只用于语义检索和人类阅读的摘要，20% 的信息丢失不影响系统功能。真正的行为数据在结构化字段里。

### 关键边界

| 场景 | 处理 |
|------|------|
| 连续 5 轮纯确定性内容 | 跳过 LLM 压缩 |
| LLM 压缩超时 | 回退到模板填充 |
| L2Summary 触发重新聚合 | 异步执行，不阻塞 |
| 双向索引 key 冲突 | 追加，不覆盖（一轮可以属于多个 L2） |

## 11. do-calculus

### 要解决什么问题

验证因果基地产出的 HARD_BLOCK 声明是否数学上自洽。不用于"发现"因果（那是键合图/Petri 网的职责），只用于"验证"。

### 算法选择：后门准则验证

**为什么是后门准则**: 后门准则是 do-calculus 三条规则中最基础、最易于工程实现的。它不要求完整的因果图满足所有马尔可夫条件，只需要检查**干预变量和结果变量之间是否存在未被控制的混杂因子**。

### 验证流程

```
输入: causal_skeleton（离线分析产出的因果 DAG）
      hypothesis（"干预 X=0 是否必然导致 Y=0?"）
Step 1: 找到 causal_skeleton 中 X 到 Y 的所有路径
Step 2: 对每条路径检查:
  - 路径上是否有混杂因子 Z 同时影响 X 和 Y?
  - 如果有，Z 是否已被控制（在 causal_skeleton 中标记为 observed）?
  - 如果存在未控制的混杂因子 → 后门准则不满足 → 不能验证 → 降级 WARN
Step 3: 如果所有后门路径都被控制 → 后门准则满足
        → do(X=0) 的效果等同于 P(Y|X=0)
        → 如果 P(Y=0|X=0) ≈ 1.0 → 验证通过 → HARD_BLOCK 成立
        → 如果 P(Y=0|X=0) < 0.95 → 验证不通过 → 降级 WARN
```

### 示例

```
骨架: {电源→电机→输出}
假设: do(电源=0) ⇒ 输出=0

路径分析:
- 电源→输出 的路径: 电源→电机→输出
- 混杂因子检查: 是否有 Z 同时影响电源和输出？无
- 后门准则满足 ✓

效果估计:
P(输出=0 | 电源=0) = 1.0 → 验证通过 → HARD_BLOCK 成立
```

### 为什么不跑完整 do-calculus 三条规则

- 规则 1（观测无关）和规则 3（工具变量）需要更完整的因果图假设
- 当前因果骨架来自离线分析（键合图/Petri 网），精度足以支持后门准则
- 如果后门准则无法验证（复杂骨架），直接降级 WARN 更安全
- 完整三条规则可以 Phase 3 再上

### 关键边界

| 场景 | 处理 |
|------|------|
| 骨架有环 | 展开为无环表示（如果可能），否则跳过验证 |
| 路径数 > 50（复杂骨架） | 限制路径深度=5，跳过超深路径 |
| 后门准则不满足 | 降级 WARN + 标记 `needs_manual_review` |
| 无骨架（新发现的因果对） | 不触发验证，等待 LightweightCausalDiscovery 产出骨架 |

## 12. 关键算法权衡总结

| 决策点 | 选了什么 | 为什么没选另一个 |
|--------|---------|----------------|
| 编译器 | LLM伪拆解+规则深挖 二合一 | 纯LLM：组合爆炸时漏约束 | 纯规则：约束消解组合爆炸 |
| 行为嵌入 | 谓词-论元原型 | 纯BGE：执行≠删除的区分差 | 动词-名词解耦：只是临时方案 |
| BehaviorGraph 权重 | EMA 更新 | 贝叶斯：不满足 IID 假设 | 简单平均：不适应行为漂移 |
| BehaviorPredictor | LLM候选+价值排序 | 序列模型：冷启动无数据 | 纯 LLM：概率估计不可靠 |
| 融合器 | 分阶段融合 | 一次性融合：延迟 150ms+ 不可接受 |
| 全局工作空间 | 简化版（100 行） | 完整 Baars：工程复杂度爆炸 |
| 噪声处理 | 先统一 0.5，攒数据再调 | 直接上自适应：没数据调什么 |
| 负知识库 | 三级+熔断 | 一刀切：特例下误杀 |
| do-calculus | 只做后门准则 | 完整三条规则：骨架精度不够 |
| 因果发现 | LightweightCausalDiscovery | 完整 Causal-Copilot：冷启动没数据 |
| δ 的初值 | 0.05，动态上调 | 固定 0.15-0.25：元角色未验证时太冒险 |
