## 不可接受的简化项（需实现）

### 1. PS.S-06: 通用规划原语库仅实现 2/17 个
- **设计文档要求**: §4.3.4 定义 17 个跨领域通用规划原语(P1-P17)，涵盖分解、分配、排序、资源、反思五维，是 Planning Skill Layer 的认知模式抽象基础
- **当前简化**: 仅实现骨架 + 2 个代表性原语，剩余 15 个原语缺失
- **为什么不可接受**: 17 个原语是"零工具场景可用"和"即插即用扩展"的核心支撑。仅 2 个原语导致系统无法处理大多数规划场景，Planning 与 Tools 正交解耦的设计哲学(ADR-003)名存实亡
- **影响**: 规划能力严重受限；新领域任务无原语可用，只能降级为纯 LLM 生成，失去结构化规划优势；混合编排引擎无法在不同模式间切换
- **实现建议**: 按优先级分批实现：P1(SequentialDecomposition), P7(SequentialFlow), P14(PlanExecuteReflect), P16(ReflectRetry) 优先；P5(ParallelMap), P8(ConditionalBranch), P9(LoopUntil) 次之；其余 Phase 2 补齐
- **涉及文件**: `ENGINEERING_PLANNING_SKILL.md` §4.3, `core/planning/` 原语库实现
- **估计工作量**: 中等（每个原语约 0.5-1 人日，首批 8 个约 1 周）

### 2. PS.S-09 / TOOL_REGISTRY 缺失: ToolShortlister 完全缺失
- **设计文档要求**: §4.6.2 要求 5 阶段漏斗筛选（标签匹配→语义排序→历史偏好→容量截断→兜底），解决 Tool Overflow 问题，默认注入 32 个工具
- **当前简化**: `ENGINEERING_TOOL_REGISTRY.md` 无 ToolShortlister 实现；`ENGINEERING_PLANNING_SKILL.md` S-09 标记为未实现
- **为什么不可接受**: 当工具数量超过 LLM 上下文窗口承载能力时，系统无筛选能力，要么超 token 预算导致 LLM 调用失败，要么手动截断导致工具选择随机，规划质量不可控
- **影响**: 工具数量>32 时系统失效；API 文档导入后工具膨胀将直接导致规划层不可用
- **实现建议**: 先实现基础版 3 阶段筛选（标签匹配→关键词重叠→容量截断），再逐步引入 embedding 语义排序和历史偏好 boost
- **涉及文件**: `ENGINEERING_TOOL_REGISTRY.md` §4.6.2 新增，或 `core/tool_registry/shortlister.py`
- **估计工作量**: 中等（3-5 人日）

### 3. PS.S-07 / TOOL_REGISTRY 缺失: ToolBindingEngine 缺失/不完整
- **设计文档要求**: §4.7 要求 4 策略绑定（精确匹配→标签匹配→语义匹配→参数兼容），将占位符绑定到实际工具，低置信度替换为 `ask_user`
- **当前简化**: `ENGINEERING_PLANNING_SKILL.md` S-07 仅实现基础绑定（精确/标签/语义），无参数兼容检查；`ENGINEERING_TOOL_REGISTRY.md` 完全缺失 ToolBindingEngine
- **为什么不可接受**: 占位符到实际工具的绑定是 Planning 与 Tools 正交解耦的关键桥梁。绑定失效意味着 LLM 生成的 `search_tool` 无法映射到 `github_api_search_repos`，工具调用失败
- **影响**: 规划层生成的 TaskGraph 无法执行；用户请求需要工具操作时系统失败率极高
- **实现建议**: 实现 4 策略绑定流水线，引入 `jsonschema` 库做参数兼容性检查；绑定置信度<0.6 时自动替换为 `ask_user` 澄清节点
- **涉及文件**: `ENGINEERING_TOOL_REGISTRY.md` §4.7 新增，或 `core/tool_registry/binding.py`
- **估计工作量**: 中等（3-5 人日）

### 4. PS.S-13: 模式回退链缺失
- **设计文档要求**: §4.5.4 要求 `MIXED → SKILL_ENHANCED → DYNAMIC → FALLBACK(ask_user)` 四级自动回退，每级记录失败原因
- **当前简化**: 仅 `replan` 按错误类型处理，无模式回退链
- **为什么不可接受**: 回退链是系统鲁棒性的核心保障。设计文档明确 ADR-007 三种模式自动选择，但选择错误时无回退机制，单次执行失败即可能终止会话
- **影响**: 执行失败率显著高于设计预期；用户因系统"卡住"而流失；MIXED 模式（合规场景）因不敢降级导致过度刚性
- **实现建议**: 在 `MixedPlanningEngine` 中实现回退状态机，每个模式执行失败时捕获异常，按降级链切换，并记录失败原因到 Telemetry
- **涉及文件**: `ENGINEERING_PLANNING_SKILL.md` §4.5.4, `core/planning/engine.py`
- **估计工作量**: 中等（3-4 人日）

### 5. PS.S-11: DynamicPlanner 多计划生成缺失
- **设计文档要求**: §4.6.3 要求生成 3 个候选计划（temperature 0.2/0.5/0.8），并基于 4 维度评分（工具存在性、参数完整性、DAG 有效性、意图覆盖）自我反思筛选最优
- **当前简化**: 未实现，仅单次 LLM 生成
- **为什么不可接受**: 单次生成无质量保障，LLM 幻觉可能导致工具名错误或参数缺失，无检测和纠正机制。自我反思是 Plan-and-Solve 和 ReAct 文献验证的核心技术
- **影响**: 规划错误率上升；无效计划直接进入执行阶段，浪费 token 和时间；Schema Guard 拦截失败时产生运行时错误
- **实现建议**: 实现 `generate_candidates()` 生成 3 个计划，然后 `evaluate_plan()` 基于 4 维度评分（无需额外 LLM 调用，规则验证即可），选择最高分
- **涉及文件**: `ENGINEERING_PLANNING_SKILL.md` §4.6.3, `core/planning/dynamic_planner.py`
- **估计工作量**: 中等（4-5 人日）

### 6. SL.S-06: 响应编排器缺失

- **为什么不可接受**: 响应编排是认知画像系统全局使用的最终出口。设计文档§7.4 明确将画像注入 L3 Response 层。固定 BALANCED 导致高元认知用户收到冗余解释、低元认知用户收到 insufficient 指导，认知增强型对话的核心目标无法实现

### 7. IP.S-04: CONFLICTING_ENTITIES 歧义检测缺失
- **设计文档要求**: §3.3.6 定义 6 种歧义类型：MISSING_ENTITY, AMBIGUOUS_ENTITY, **CONFLICTING_ENTITIES**, VAGUE_SCOPE, UNSUPPORTED_OPERATION, MULTIPLE_INTENTS
- **当前简化**: 仅实现 5 种，缺少 CONFLICTING_ENTITIES（实体冲突检测）
- **为什么不可接受**: 实体冲突（如用户同时提供"PID=1234"和"进程名=chrome.exe"但指向不同进程）会导致执行错误。缺少该检测意味着系统可能基于矛盾实体生成错误 TaskGraph，执行不可预测的结果
- **影响**: 实体矛盾场景下的执行错误率上升；用户信任度下降；调试困难（错误原因难以归因）
- **实现建议**: 在 `AmbiguityDetector` 中增加 `CONFLICTING_ENTITIES` 类型检测，基于规则检查同类型实体值冲突（如两个不同 PID）、跨类型语义冲突（如 PID 与进程名不匹配）
- **涉及文件**: `ENGINEERING_INTENT_PARSER.md` §5.6, `core/intent_parser/ambiguity.py`
- **估计工作量**: 低（1-2 人日）

### 8. IP.S-07: EntityCache 写入未激活
- **设计文档要求**: §3.3.7 要求上下文合并时将高置信度实体(≥0.8)写入跨轮 `_entity_cache`，供下一轮参照消解使用
- **当前简化**: `_entity_cache` 字段已定义，但 `add_intent()` 仅写入 `resolved_entities`（单值字典），未写入 `_entity_cache`（列表缓存）
- **为什么不可接受**: 参照消解（Reference Resolver）是八阶段流水线的第 2 阶段，依赖实体缓存进行代词替换。缓存不激活意味着"这个地址"、"那个值"等引用无法解析，多轮对话连续性严重受损
- **影响**: 多轮对话中实体继承失败率极高；用户需重复输入已提供信息；对话体验断裂
- **实现建议**: 在 `ContextMerger` 中补充 `_update_entity_cache()` 调用，将 `confidence >= 0.8` 的实体追加到 `_entity_cache`；在 `ReferenceResolver` 中接入 `_entity_cache` 读取路径
- **涉及文件**: `ENGINEERING_INTENT_PARSER.md` §5.8, `core/intent_parser/context_merger.py`, `reference_resolver.py`
- **估计工作量**: 低（1-2 人日）

### 9. PS.S-08: SchemaGuard + ToolExecutor 不完整
- **设计文档要求**: §4.8 要求完整 JSON Schema 验证（必填参数、类型、枚举值）和 3 后端执行分发（LOCAL_FUNCTION/HTTP_API/MCP_REMOTE）
- **当前简化**: 仅基础验证层 + 分发接口骨架
- **为什么不可接受**: 工具参数错误可能导致严重后果（如错误内存地址写入、错误 API 调用）。Schema Guard 是安全执行的最后防线。骨架实现无法阻止参数类型错误和非法枚举值
- **影响**: 工具执行失败率上升；错误参数可能触发外部系统异常；安全边界收缩
- **实现建议**: 引入 `jsonschema` 库实现完整参数验证；为 3 种后端实现执行适配器（本地函数直接调用、HTTP 构造 requests、MCP 构造 JSON-RPC）
- **涉及文件**: `ENGINEERING_PLANNING_SKILL.md` §4.8, `core/tool_registry/executor.py`
- **估计工作量**: 中等（3-4 人日）

### 10. PS.S-12: PlanningSkill 三级详细度简化
- **设计文档要求**: §4.4.3 要求三级详细度枚举（SKELETON/STANDARD/DETAILED），控制 LLM 自由度和流程完整性
- **当前简化**: `level: str` 简化字符串，无枚举约束
- **为什么不可接受**: 三级详细度是 MIXED 模式（合规场景）的核心控制机制。DETAILED 级别要求骨架不可修改，若用字符串表示，无法保证混合编排引擎正确识别"不可修改"步骤，合规流程完整性受损
- **影响**: 金融/医疗等合规场景下流程可能被 LLM 擅自修改；Skill 模板的约束力度不可控
- **实现建议**: 定义 `SkillLevel` 枚举（Enum），在 `MixedPlanningEngine` 中根据枚举值执行不同约束策略（DETAILED=骨架不可改，SKELETON=仅参考）
- **涉及文件**: `ENGINEERING_PLANNING_SKILL.md` §4.4.3, `core/planning/skill.py`
- **估计工作量**: 低（1 人日）

### 11. PS.S-10: 认知画像联动未实现
- **设计文档要求**: §4.5.3 模式选择决策树要求融入认知画像（高发散→MIXED 降级为 SKILL_ENHANCED，低 g→DYNAMIC 升级为 SKILL_ENHANCED）
- **当前简化**: 未实现，模式选择仅基于 Skill 匹配度
- **为什么不可接受**: 设计文档 ADR-007 明确三种模式自动选择基于认知画像。缺失导致所有用户收到相同规划策略，违反"基于用户认知模型的持续推断与自适应"核心设计哲学
- **影响**: 高发散用户在 MIXED 模式下感到过度约束；低元认知用户在 DYNAMIC 模式下迷失；规划体验非个性化
- **实现建议**: 实现 `_select_mode_with_profile()` 函数，读取 `CognitiveProfile` 的 divergence/metacognition/g 因子，覆盖默认决策树
- **涉及文件**: `ENGINEERING_PLANNING_SKILL.md` §4.5.3, `core/planning/engine.py`
- **估计工作量**: 低（2 人日）

### 12. SL.S-07: 核心 REST 端点不完整
- **设计文档要求**: §6.4 要求完整 REST API：/chat, /parse, /execute, /session
- **当前简化**: 仅实现部分管理端点（/health, /sessions/{id}, /metrics），缺少 /chat, /parse, /execute 核心业务端点
- **为什么不可接受**: 虽然 WebSocket 已覆盖实时交互，但 /parse 和 /execute 是外部系统调用和调试的关键接口。REST API 的缺失限制了系统的可集成性和可测试性
- **影响**: 外部工具无法直接调用解析和执行能力；前端测试和自动化集成困难
- **实现建议**: 补充 `/chat`（同步对话）、`/parse`（仅意图解析）、`/execute`（仅工具执行）端点，内部复用现有 Orchestrator 逻辑
- **涉及文件**: `ENGINEERING_SERVICE_LAYER.md` §6.4, `core/service/api.py`
- **估计工作量**: 低（2-3 人日）

### 13. IP.S-05: 上下文评分简化实现
- **设计文档要求**: §3.3.4 要求 `context_score` 占 10% 权重，基于历史意图相似度计算（追踪深度调节）
- **当前简化**: 仅检查 `tracking_depth > 0.6` 时无操作，未实现完整历史相似度计算
- **为什么不可接受**: 上下文评分是意图分类的关键调优因素，特别是在多轮对话中维持话题连续性。10% 权重的缺失导致跨轮意图切换时分类精度下降
- **影响**: 多轮对话中意图分类准确率下降；话题连续性维护能力削弱
- **实现建议**: 实现 `history_similarity()` 函数，计算当前意图与最近 N 轮意图的类别重叠和实体重叠，映射到 [0,1] 的 context_score
- **涉及文件**: `ENGINEERING_INTENT_PARSER.md` §5.4, `core/intent_parser/classifier.py`
- **估计工作量**: 低（1-2 人日）

### 14. IP.S-06: 复合意图深度分解简化
- **设计文档要求**: §3.3.5 要求复合意图（如"扫描并修改"）的深度语义分解，而非仅按连词分段
- **当前简化**: 仅按连词（"然后"、"and then"）分段浅层分解
- **为什么不可接受**: 浅层分解无法处理隐含复合意图（如"帮我看看这个内存地址然后改成0xFF"中的"看看→读取→修改"链），导致 TaskGraph 构建不完整，遗漏关键步骤
- **影响**: 复合意图场景下的执行覆盖率下降；用户需分多次表达完整意图
- **实现建议**: 在连词分段基础上，增加语义分析：检查动词链（"看看→修改"映射为 READ→MODIFY），识别隐含子意图
- **涉及文件**: `ENGINEERING_INTENT_PARSER.md` §5.5, `core/intent_parser/multi_intent.py`
- **估计工作量**: 中等（2-3 人日）

### 15. CM.S-05: 语义回热缺失
- **设计文档要求**: §5.3.5 要求 Cold 层恢复基于语义相似度，处理隐式引用（"之前讨论的那个"）
- **当前简化**: 仅基于 `topic_id` 精确匹配回热
- **为什么不可接受**: 用户很少使用精确 topic_id 引用历史。精确匹配无法处理模糊引用，导致跨主题上下文恢复失败，长期对话的上下文继承能力严重受限
- **影响**: 长会话中用户提及历史话题时系统无法恢复上下文；用户感知"系统忘记了之前的对话"
- **实现建议**: 引入轻量 embedding 模型（3B 参数）或关键词匹配，计算用户输入与 Cold 层主题标签的语义相似度，相似度>阈值时回热
- **涉及文件**: `ENGINEERING_CONTEXT_MANAGER.md` §5.3.5, `core/context_manager/window.py`
- **估计工作量**: 中等（3-4 人日）

### 16. TT.S-04: 主题权重衰减缺失
- **设计文档要求**: §5.2.2 要求 EMA 更新，非活跃节点权重衰减
- **当前简化**: 仅更新活跃主题权重，不活跃主题不衰减
- **为什么不可接受**: 不衰减导致 Topic Tree 无限增长，内存占用持续膨胀，权重分布失真（旧话题与新话题权重无区分），主题切换检测阈值失效
- **影响**: 长会话内存泄漏；主题切换检测准确率下降；系统响应延迟随树规模增长而增加
- **实现建议**: 在 `TopicTree` 中增加定时衰减任务（或惰性检查），每轮遍历所有节点，非活跃节点权重按 `W *= decay_factor` 衰减
- **涉及文件**: `ENGINEERING_TOPIC_TREE.md` §5.2.2, `core/dialogue/topic_tree.py`
- **估计工作量**: 低（1-2 人日）

### 17. CM.S-02: Token 精确计数缺失
- **设计文档要求**: §5.3.4 要求自适应窗口大小基于 `TokenBudget`；确保不超出 LLM 上下文限制
- **当前简化**: 使用字符数/4 粗略估算
- **为什么不可接受**: 中英文混合场景下字符数/4 估算误差可达 30-50%，可能导致实际 token 数超预算，LLM 调用失败或上下文被截断，关键信息丢失
- **影响**: 上下文窗口超预算风险；LLM 调用失败率上升；重要历史信息被截断
- **实现建议**: 引入 `tiktoken` 或轻量 tokenizer，实现精确 token 计数；如依赖不可接受，可基于语言检测使用不同的字符→token 换算系数
- **涉及文件**: `ENGINEERING_CONTEXT_MANAGER.md` §5.3.4, `core/context_manager/token_counter.py`
- **估计工作量**: 低（1 人日）

### 18. LL.S-02: JSON Schema 验证缺失
- **设计文档要求**: §4.8.1 要求 Schema Guard 验证 JSON Schema（参数类型、枚举值、必填项）
- **当前简化**: `json_schema` 字段已定义但无验证逻辑
- **为什么不可接受**: LLM 生成的结构化输出（工具参数）无验证，错误类型或非法枚举值直接进入执行层，导致工具调用失败或副作用。这是 Schema Guard 的底层依赖
- **影响**: 工具参数错误率上升；执行失败难以归因；安全边界收缩
- **实现建议**: 引入 `jsonschema` 库，在 `LLMProvider.generate()` 返回后验证输出是否符合请求时的 `json_schema`
- **涉及文件**: `ENGINEERING_LLM_PROVIDERS.md` §4.8, `core/llm/providers/base.py`
- **估计工作量**: 低（1-2 人日）

### 19. ML.S-01: LLM 引擎实例化代码未实现
- **设计文档要求**: §2.2, §3.2 要求 6 个专用 LLM 实例（PCR-LLM, Intent-LLM, Planning-LLM, Skill-LLM, Reflective-LLM, Companion-LLM）
- **当前简化**: Prompt 模板已定义，底层运行代码、错误处理、重试逻辑未实现
- **为什么不可接受**: 认知双工（Hybrid Engine）依赖 PCR-LLM 和 Intent-LLM 的并行运行。底层代码未实现意味着 Hybrid Engine 仅为设计，无法实际运行，PCR 和 Intent Parser 的 LLM 增强路径不可用
- **影响**: 系统退化为纯规则引擎，复杂场景解析能力大幅下降；LLM 的噪声过滤、歧义消解、期望推断能力无法发挥
- **实现建议**: 基于 `LLMProvider` 基类，实现 6 个专用子类，每个加载对应 Prompt 模板，配置不同温度参数和重试策略
- **涉及文件**: `ENGINEERING_MULTILAYER_LLM.md` §5.3, `core/llm/engines/`
- **估计工作量**: 中等（5-7 人日）

### 20. TR.S-02: OpenAPI 导入未实现（需确认集成状态）
- **设计文档要求**: §4.6.1 要求通过 APIDocPreprocessor 解析 OpenAPI/Swagger 后自动注册工具
- **当前简化**: `ENGINEERING_TOOL_REGISTRY.md` 标记为接口预留；`ENGINEERING_API_DOC_PREPROCESSOR.md` 已独立实现解析功能

## 核查总结

| 类别 | 简化项总数 | 不可接受 | 可接受 | 不可接受占比 |
|------|-----------|---------|--------|-------------|
| Layer 0 (PCR) | 5 | 1 | 4 | 20% |
| Layer 1 (Intent Parser) | 8 | 5 | 3 | 62.5% |
| Layer 1.5 (Planning) | 13 | 8 | 5 | 61.5% |
| Layer 2 (Dialogue/State) | 15 | 4 | 11 | 26.7% |
| Layer 3 (Service) | 8 | 2 | 6 | 25% |
| 横切: 认知画像 | 5 | 0 | 5 | 0% |
| 横切: 记忆系统 | 2 | 0 | 2 | 0% |
| 横切: 可观测性 | 5 | 0 | 5 | 0% |
| 工具/API/LLM/其他 | 22 | 3 | 19 | 13.6% |
| **合计** | **83** | **23** | **60** | **27.7%** |

**关键结论**:

1. **Planning Skill Layer 是风险最高的模块**：13 个简化项中有 8 个不可接受，核心原语库、ToolShortlister、ToolBindingEngine、模式回退链、多计划生成等关键组件缺失或严重不完整。这是设计目标无法达成的最大风险点。

2. **Intent Parser 层多轮能力受损**：EntityCache 未激活、上下文评分简化、CONFLICTING_ENTITIES 缺失、复合意图浅层分解，导致多轮对话连续性和解析鲁棒性下降。

3. **服务层响应编排是用户体验瓶颈**：固定 BALANCED 响应格式使认知画像系统的全局使用无法产生用户感知价值，核心设计哲学"认知增强型对话"受损。

4. **跨文档不一致需优先修复**：`ENGINEERING_TOOL_REGISTRY.md` 和 `ENGINEERING_SERVICE_LAYER.md` 的等价性检查声称"全部覆盖"，但实际缺失核心组件，这种**不诚实标记比功能缺失本身更危险**——它掩盖了真实差距，导致后续决策基于错误信息。

5. **Phase 1 可接受的基础简化**：数据契约类型安全(字符串/枚举)、基础设施(Redis/PostgreSQL/负载均衡)、多模态、性能缓存、高级安全特性等，属于合理的阶段性简化，不影响核心设计目标。
