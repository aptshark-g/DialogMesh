## 1. 整体叙事脉络

### 第一天（7月4日）：从行为推演到因果基地

**起点**——你（刘子健）指出 DialogMesh v3.0 的问题："对话树的核心不是分类，而是建模行为因果。用户可以加一个训练——LLM 做猜测，成功了加分，失败了用户纠正就是修正信号。"

这触发了整个 v3.1 的设计。核心判断：**Topic Tree 只有主题切分，没有行为推演链。对话树不应该只是"用户聊过什么"，而是"用户在这个话题下做了什么操作、操作之间有怎样的因果概率"。**

v3.1 的设计提出三个核心改造：
- **行为推演树**：TopicTreeNode 新增三链（行为链/因果链/关联链）
- **双层摘要系统重构**：一级摘要（结构化快照）+ 二级摘要（行为图聚合）
- **在线训练闭环**：预测→验证→修正→回流全流程

你在此基础上提出了三个优化方向：World Model 模拟推演、动态有损压缩（熵驱动）、神经符号约束注入。讨论的结果是全部做轻量化：World Model 降级为 Skill 结果预测，LLM 压缩从硬约束改为软目标，神经符号约束只做 Schema Guard。

> **当时的核心判断**："想做全量的都不做全量。理论正确性让位于工程可行性。"

然后是 Literature Cortex 的交叉分析。你分享了那份文档，我们发现了两个系统在架构思路上惊人的重叠——五层控制流、锚点硬闸门、成本分级。从中提取了三个可借鉴的设计：BudgetScheduler（成本控制）、负知识库（永久跳过错误路径）、ExecutionMode 枚举（L5 模态切换）。

接着你抛出了一个更深的洞见："约束本身超越了某种性质即可忽略其关联性变成因果性"。这个观点触发了因果基地的设计。核心判断是：**因果不是世界的固有属性，而是认知主体在约束空间中做出的投射。当一组约束的稳定性超过观测者的参考框架时，这组约束定义的关系就是"足够好的因果"。**

这个判断的意义在于：因果基地的 `structural_prior` 不应该输出 0/1 的二元判断，而应该输出**约束稳定性评分**——键合图说的事更可信（0.95），LLM 推断的事需要验证（0.3-0.5）。

### 第二天（7月5日）：从理论深化到工程落地

你提出了五个理论升级方向：SCM do-calculus、全局工作空间、Q-learning 优化、神经符号约束、谓词-论元嵌入。

讨论的结论是：
- **三个立即采纳**：do-calculus 做 HARD_BLOCK 验证、简化版全局工作空间（100行）、谓词-论元嵌入替代动词-名词解耦
- **两个做轻量版**：Q-learning 改为四维价值评估排序，流式符号约束做增量验证而非 logit 层
- **一个暂不入**：统一神经符号表示空间（等学术界成熟）

全部整合进 v3.3 设计文档。

然后你让我补充了算法实现设计文档——每个模块用什么算法、为什么选它、输入输出是什么、关键边界在哪。12 章，覆盖编译器、谓词-论元嵌入、BehaviorGraph、Predictor、Rewarder、分阶段融合器、元角色三步映射、负知识库、FoA、L1Summary、do-calculus。

## 2. 思维转折点（那些改变方向的时刻）

### 转折 1: "约束让关联变成因果"

这是你（liuzijian）提出来的。你用了洗车店的例子：洗车店只有 5 米，但"要开车吗？"答案是**需要**——因为"洗车"的宾语是车，"洗车店"隐含车必须到场，"开"暗示小汽车量级。约束连起来：要洗车→车到店→车怎么到店→开过去。即使 5 米也需要开。

这个例子的力量不在于"LLM 搞错了"，而在于它揭示了**所有的因果本质上都是在约束空间内的稳定关联**。

**这个判断改变了我们对因果基地的定位**：因果基地不是说"物理定律=因果"，而是说"物理定律是在极其稳定的约束空间内成立的关联，因此在这套约束下可以被当作因果使用"。反过来，键合图证明的关系也可能在更宽的约束空间下失效——比如考虑量子效应时，经典力学就不再是因果。

这直接导致了 `structural_prior` 的设计修正：永远不说 1.0，永远不上限，永远以"约束稳定性"而非"因果必然性"来表述。这是一个哲学判断落到工程参数上的典型案例。

### 转折 2: "三步映射"——从一步映射到约束条件中介

在 v3.2 因果基地的设计初稿里，元角色映射是**一步的**：自然语言行为描述 → 直接映射到元角色骨架。你（或者我们在讨论中共同意识到了）这个风险：一步映射出错，`structural_prior` 就从"先验"变成了"错误的必然性"。

修正为三步：**自然语言 → 约束条件 → 候选结构 → 匹配度排序**。

约束条件作为中间层，即使有偏差也只缩小搜索空间，不直接污染映射结果。这一步修正本质上是一个**容错升级**——我们承认第一步（自然语言理解）不可能 100% 正确，但可以通过在错误和不精确之间插入一个约束条件层来限制错误的传播半径。

### 转折 3: "δ 从 0.05 开始，不做固定值"

在最初的 BehaviorEdge 权重公式里，`δ × structural_prior` 的 δ 被设为固定的 0.15-0.25。这意味着不管元角色映射的质量如何，结构因果都占据 15-25% 的权重。

这个设计有问题：元角色映射刚上线时质量未知，给一个固定的高权重是危险的。修正方案是**δ 从 0.05 开始，每个 Reflective 周期根据 `correction_count` 验证元角色映射的准确性，准确的行为对上 δ 逐渐上调，不准确的维持 δ=0.05。**

这是一个"先信任规则、后信任自身经验"的工程哲学——而原始方案是"先信任自身经验、后让规则补充"。

### 转折 4: 融合器的"分阶段"——不只是算法优化

最初融合器设计是"四路结果全部返回→一次性融合"。工程落地分析触发了修正：Track-0（算法 10ms）、Track-1（LLM 150ms）、Track-P（预测 80ms）、因果基地（<1ms）——等齐了才融合意味着 150ms 硬延迟。

但"等齐了再融合"不仅是一个性能问题，它是一个**架构问题**——它假设四条轨道的信息是同质的、可以一次性合并的。实际上，算法和因果基地的信息是确定性的、<10ms 就可以部署的硬事实；LLM 的语义理解是软信息、需要 150ms。硬事实不应该等软信息才生效。

修正为分阶段融合：T=10ms 硬事实部署 → T=80ms 预测注入（不影响已部署的硬事实） → T=150ms 语义补充（有可能修正但只在冲突时）。这意味着系统在 10ms 就有一个可用输出，而不是等到 150ms 才知道回答什么。

## 3. 达成共识的关键判断

### 判断 1: 因果基地不做实时推理

四个形式化引擎（键合图/Petri网/系统动力学/元角色映射）每个都是 50-500ms 的 Heavy 操作。实时对话延迟预算 <200ms。这两条之间没有妥协空间——**离线预计算 + 在线只读缓存**是唯一可行的方案。

这意味着因果基地的角色定义是**先验知识源**而不是**推理引擎**。它告诉 BehaviorGraph"根据物理/逻辑/数学，A 和 B 之间应该存在什么关系"，但不在每次用户输入时重新推导这个关系。

### 判断 2: 约束补全不替代 LLM，而是限制 LLM 最大的弱点

LLM 做约束消解的最大问题是**注意力竞争导致的弱约束压制**——"5米"这种强信号抢走所有注意力，弱约束（"洗车需要车到店"）被压制。规则引擎做约束消解的最大问题是**组合爆炸**——10 个维度 × 20 个候选值 = 200 种组合。

**二合一**（LLM 粗切割 + 规则只在 LLM 不确定的维度上深挖）同时避开了这两个问题：LLM 不做约束消解（只做粗粒度语义角色标注），规则不做全量消解（只针对 LLM 不确定的 1-2 个维度）。组合爆炸被**选择性深挖**绕过了。

### 判断 3: 神经符号约束不做 logit 层，做结构化输出增量验证

NeSyS 的 logit 约束在数学上优雅，但工程上有两个致命问题：（1）不同 LLM 调用场景需要不同的约束集，约束编译可能比 LLM 调用还慢；（2）对多 token 约束（如"PID=1234"）无效——1234 是 4 个 token，不能在单个 token 的 logit 层做约束。

修正为流式结构化输出增量验证：LLM 流式输出 JSON，每条 key-value 出来时立即对照约束规则库。比 logit 层慢 <5ms，但工程可行且支持多 token 约束。

### 判断 4: 融合器加上简化的全局工作空间（100 行就够了）

完整实现 Baars 的全局工作空间理论需要并行处理器竞争广播+意识闸门，工程复杂度极高而且对我们场景不必要。简化版只需要在现有融合器上加 `track_repression_count` 字典 + 动态优先级调整——核心逻辑不到 100 行。

这个判断的底层逻辑是：**我们不是在做"机器意识"研究，我们是在做一个对话系统的多轨融合机制。Bars 理论的启发足够我们解决"某个轨道一直被压制"的公平性问题，不需要把整个理论照搬过来。**

### 判断 5: 谓词-论元嵌入替代动词-名词解耦

纯 BGE-small 嵌入下"运行程序"和"删除程序"的余弦相似度 0.78——这是个灾难。动词-名词解耦（`vec = verb_vec × 0.7 + noun_vec × 0.3`）是缓解而非根治——它没有区分 execute 和 delete 这两个在逻辑上截然不同的谓词。

谓词-论元嵌入用谓词类（execute/delete/analyze/...）提供硬边界：谓词类不同 = 根本不同的行为，不论论元多相似。论元（Program/Log/Config/...）保留 BGE-small 语义信息。规则 <50 条，一次性工作，覆盖常见行为类型。

### 判断 6: 噪声自适应先不做，先跑数据

你在设计文档里提出了"噪声自适应训练"——高噪声用户的信号不应该被过滤，而是降低单轮权重、拉长观察窗口。方向是对的，但"什么是噪声"在没有真实数据时无法定义。

我们的决定：**Phase 1 先统一 `noise_level=0.5`，攒 2-4 周真实数据后回来标注分析。** 三件事可以区分信号和噪声：（1）纠正后是否在 3 轮内被再次纠正（自我矛盾=噪声）；（2）纠正后是否持续沿用新路径（有效学习=信号）；（3）纠正文本中是否有否定词（强化信号）。

### 判断 7: L1Summary 的 20% 信息丢失是软目标，不是硬约束

在三级自适应摘要引擎里，确定性内容（工具输出/代码）用规则引擎零丢失；半确定性内容（明确指令）用模板填充 ≤10% 丢失；非确定性内容（模糊需求/情感）用 LLM 压缩，≤20% 是**软目标**——不一定能达到，也不需要达到，因为行为数据已经结构化保存了。

核心洞察：**行为和因果信息在 `meta_info` 结构化字段中是零丢失的。`core_semantics` 字段只用于语义检索和人类阅读的摘要。LLM 压缩 20% 的丢失不影响系统功能。**

### 判断 8: δ 从 0.05 开始动态上调，不做固定值

元角色映射的质量在 Phase 1 是未知的。给 `structural_prior` 一个固定的 15-25% 权重是冒险的——如果元角色映射质量不好，这个 δ 不是"无害的错误先验"，而是"系统性地把 BehaviorGraph 往错误方向推"。

δ 从 0.05 开始，每个 Reflective 周期验证：如果 `structural_prior > 0.3` 且 `correction_count` 低（用户不纠正这条边），上调 δ 至 0.15。如果 `correction_count` 高，维持 δ=0.05 或下调。**这是"系统对自己的元角色映射能力持怀疑态度，直到数据证明它是对的"。**

## 4. 我们放弃过什么

| 放弃的方案 | 原因 | 替代方案 |
|-----------|------|---------|
| 全量 World Model（状态推演） | LLM 成本爆炸，数据需求大 | Skill 元数据 + 历史统计做结果预测 |
| LLM 摘要 ≤20% 硬约束 | LLM 生成不可靠，无法保证 | 三级自适应策略，≤20% 改为软目标 |
| Q 网络完整训练 | 冷启动时期样本不足数百轮 | 四维价值评估排序（概率+成功率+认知负载+画像匹配） |
| NeSyS logit 层约束 | 约束编译复杂，多 token 约束失效 | 流式结构化输出增量验证 |
| 完整全局工作空间（Baars） | 过度设计，工程代价巨大 | 融合器加 simplify workspace（100 行） |
| 统一神经符号表示空间 | 学术未成熟，工程不可用 | 谓词-论元嵌入做工程最优解 |
| 因果基地实时推理 | 不可行，50-500ms | 离线预计算 + 在线只读缓存 |
| 因果基地 δ 固定权重 | 元角色映射质量未知 | δ 从 0.05 动态上调 |
| `structural_prior` 说"确定性因果" | 三步映射可能出错 | 改为"约束稳定性评分"，上限 0.7 |
| Causal-Copilot 全量集成 | 太重了，子 Agent 嵌套 | Reflective 层做轻量因果发现 |
| 噪声自适应上线即用 | 无真实数据时无法定义"噪声" | Phase 1 统一 noise_level=0.5，攒 2-4 周数据后标注 |
| 完整 do-calculus 三条规则 | 骨架精度不支持，规则 1/3 需要完整因果图 | 只做后门准则验证 |

**核心模式**："理论正确但工程太重 → 做轻量版"。这在决策中反复出现，不是一个孤立的判断而是一个系统性的模式——我们几乎在所有"理论深度"和"工程可行性"的交叉点上，都选择了先落地、再深化。

## 5. 核心设计哲学

### 哲学 1: 因果不是发现出来的，是投射出来的

> "因果不是世界固有的属性，而是认知主体在约束空间中做出的投射。当一组约束的稳定性超过观测者的参考框架时，这组约束定义的关系就是'足够好的因果'。"

这不是哲学空话——它直接决定了 `structural_prior` 的设计：从不输出 1.0，永远以"约束稳定性"而非"因果必然性"来表达。

### 哲学 2: 结构因果补充统计因果，不是替代

我们不是用因果基地替代 BehaviorGraph 的统计学习——而是补充它。统计因果（LLM 推理+历史频次）处理"常见模式"，结构因果（键合图/Petri 网/元角色）处理"确定性关系"。两者不是竞争，是互补。

这体现在 BehaviorEdge 的权重公式里：`w = α×LLM概率 + β×频率 + γ×画像 + δ×结构先验 + ...`。δ 的加入不是让结构因果凌驾于统计因果之上，而是给统计因果一个先验骨架——当统计因果没有数据时（冷启动），结构因果提供方向；当统计因果有大量数据时，结构因果被稀释。

### 哲学 3: LLM 不做约束消解，LLM 只做粗切割

这是约束补全编译器的设计基础。LLM 擅长"大概是什么"，不擅长"为什么必须是这个"。规则擅长"必须是这个"，但不擅长"有哪几种可能"。

让 LLM 做语义角色标注（粗切割），让规则做约束消解（只在 LLM 不确定的维度上）。两者各取所长。

### 哲学 4: 纠错即训练——反向信号比正向信号更有价值

用户默默接受系统的建议 → 低价值信号（可能是用户没注意、没反馈、或只是接受）。用户明确纠正 → 极高价值信号（系统犯错的事实证据、正确路径的标注）。

这体现在 BehaviorRewarder 的奖励规则里：预测命中 +0.10，用户纠正 -0.20——**纠正的惩罚是预测命中的两倍**。而且还加了一个"快速纠正通道"：连续纠正 2 次 → 跳过 EMA 平滑直接硬覆盖。

### 哲学 5: 先信任规则，后信任自身经验

系统在没有数据时的姿态是"怀疑自己"。δ 从 0.05 开始、不固定。`structural_prior` 上限 0.7 而非 1.0。负知识库有三次熔断——用户坚持 3 次后系统学习新上下文规则。噪声自适应不上线先跑数据。

所有这些判断共享同一个底层逻辑：**系统对自己的形式化知识（因果基地、元角色、约束规则）持谦逊态度。这些知识的权威性不是凭空赋予的，而是通过用户数据逐步验证的。**

---

## 6. 你应该记住的最重要的几个设计决策

1. **行为链 • 因果链 • 关联链三链在 TopicTreeNode 中并行**，不在独立的行为树上——避免跨树查询的复杂化。

2. **BehaviorGraph 是独立的图引擎**（存权重），TopicTreeNode 是行为链的拓扑索引（存节点+边 ID）。两者通过 edge_id 引用。

3. **因果基地离线跑，在线只读**——这句话是整个因果基地设计的基石。所有 Heavy 操作（键合图/Petri 网/系统动力学/元角色映射）只在会话间隙执行。

4. **融合器不是一次性融合，是分阶段（10ms→80ms→150ms）**——这决定了系统在 10ms 就有可用输出，LLM 挂了也不影响核心功能。

5. **元角色映射是三步（约束条件中介），不是一步**——这决定了映射出错时错误传播的半径被约束条件限制。

6. **负知识库有三次熔断**——用户坚持 3 次后系统学习新的上下文规则。从"纯水不导电"到"但电解时导电"。

7. **ABL 反思向量不影响权重，写入 Cognitive Tree 供 Meta-Cognitive 分析**——数值修正归 BehaviorRewarder，可解释性归反思向量。

8. **谓词-论元嵌入的谓词类提供硬边界**——"运行程序"和"删除程序"在谓词类层面就分开了，不依赖嵌入向量碰巧区分。

9. **do-calculus 只做验证（后门准则），不做发现**——因果骨架是离线分析产出的，do-calculus 验证它自洽还是自相矛盾。

10. **约束稳定性取代因果必然性**——`structural_prior` 不说"这是因果"，说"在当前的约束空间里，这个关系的稳定性是 X"。
